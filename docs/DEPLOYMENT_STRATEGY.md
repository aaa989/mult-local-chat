# AI Agent 中台系统 - 部署策略说明

---

## 一、项目概述

本项目是一个基于大语言模型的智能 Agent 中台系统，旨在提供综合问答、代码生成、视觉 OCR、语音识别等多种 AI 能力。项目最初计划依托服务器端部署大模型，但在实践过程中遇到了诸多技术挑战，最终采用**"优先服务器调用 + 保底本地部署"**的混合部署方案。

---

## 三、服务器端部署遇到的问题与挑战

### 3.1 功能完整性问题

**问题描述：**
- **OCR 功能缺失**：vLLM 框架主要针对纯文本大语言模型设计，无法直接部署视觉语言模型（如 minicpm-v）
- **语音识别缺失**：需要额外部署语音识别服务，增加系统复杂度

**影响评估：**
- 无法满足项目对多模态能力的需求
- 需要额外开发和维护多个独立服务

### 3.2 响应延迟问题

**问题描述：**
- 服务器端模型推理存在网络传输延迟
- 并发请求时响应时间明显增加
- 模型加载时间较长（约 30 秒以上）

**性能测试数据：**
| 请求类型 | 平均响应时间 |
|----------|------------|
| 简单问答 | 3-5 秒 |
| 复杂推理 | 10-15 秒 |
| 代码生成 | 15-20 秒 |

### 3.3 资源占用问题

**问题描述：**
- Qwen2.5-7B 模型显存占用约 15GB+
- 服务器资源有限，无法同时部署多个模型
- 多用户并发访问时性能下降明显

### 3.4 OCR 识别效果问题

**问题描述：**
- 在服务器端测试 OCR（使用 rapidocr）时，识别效果较差
- 识别结果存在大量错误（如图中 "Reached"、"maxim" 等错误识别）
- 中文识别准确率低

## 四、本地部署方案（Ollama）

### 4.1 技术选型

| 组件 | 选型 | 理由 |
|------|------|------|
| 模型管理 | Ollama | 轻量级、一键部署、支持多模型 |
| 聊天模型 | DeepSeek R1 1.5B | 轻量化、中文支持好、推理快 |
| OCR 模型 | minicpm-v | 支持多模态、可通过 Ollama 部署 |
| 语音识别 | faster-whisper | 本地运行、支持中文、速度快 |

### 4.2 部署架构

```
┌─────────────────────────────────────────────────────────────┐
│                     AI Agent 中台系统                       │
├─────────────────────────────────────────────────────────────┤
│  前端 (React)  ←→  后端 (FastAPI)  ←→  Ollama (本地模型)   │
│                   ↓                                        │
│            faster-whisper (语音识别)                        │
└─────────────────────────────────────────────────────────────┘
```

### 4.3 模型配置

| 模型名称 | 来源 | 功能 |
|----------|------|------|
| deepseek-r1:1.5b | Ollama | 综合问答、代码生成、研究分析 |
| minicpm-v | Ollama | 视觉 OCR、图像理解 |
| faster-whisper-small | Hugging Face | 语音转写、语义分析 |

### 4.4 本地部署优势

| 对比项 | 服务器端部署 | 本地部署 |
|--------|-------------|----------|
| **功能完整性** | 仅支持文本 LLM | 支持多模态（OCR、语音） |
| **响应速度** | 网络延迟 + 推理时间 | 本地推理，响应更快 |
| **资源占用** | 高（15GB+ 显存） | 低（4GB 显存即可运行） |
| **部署复杂度** | 高（需服务器权限） | 低（一键部署） |
| **数据隐私** | 数据需上传服务器 | 数据本地处理，更安全 |
| **离线使用** | 依赖网络 | 完全离线可用 |

## 五、混合部署方案（推荐）

### 5.1 架构设计

采用**"优先服务器调用 + 保底本地部署"**的混合策略：

```
┌─────────────────────────────────────────────────────────────┐
│                   请求路由层                                │
│                    ↓                                        │
│    ┌────────────────┐          ┌────────────────┐          │
│    │  服务器端模型   │ ←失败→  │   本地模型     │          │
│    │  (Qwen2.5-7B)  │          │ (DeepSeek R1) │          │
│    └────────────────┘          └────────────────┘          │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 策略说明

| 场景 | 调用策略 | 说明 |
|------|---------|------|
| **综合问答** | 优先服务器 | 大模型效果更好 |
| **代码生成** | 优先服务器 | 代码质量更高 |
| **OCR 识别** | 本地部署 | 服务器不支持 |
| **语音识别** | 本地部署 | 服务器不支持 |
| **服务器不可用** | 自动切换本地 | 故障兜底 |
| **网络不可用** | 自动切换本地 | 离线可用 |

### 5.3 实现方式

在后端 `main.py` 中实现智能路由：

```python
async def get_completion(message: str, use_server: bool = True) -> str:
    """智能路由：优先服务器，保底本地"""
    if use_server:
        try:
            # 尝试调用服务器端模型
            result = await call_server_model(message)
            return result
        except Exception as e:
            # 服务器调用失败，切换本地
            logger.warning(f"服务器调用失败: {e}，切换本地模型")
    
    # 本地模型兜底
    return await call_local_model(message)
```

## 七、运行方法

### 7.1 环境准备

```powershell
# 1. 安装 Ollama
# 下载地址：https://ollama.com/download/windows

# 2. 下载模型
ollama pull deepseek-r1:1.5b
ollama pull minicpm-v

# 3. 安装 Python 依赖
cd agent-hub-backend
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt

# 4. 安装前端依赖
cd agent-hub-frontend
npm install
```

### 7.2 启动服务

**启动 Ollama（首次运行自动启动）：**
```powershell
ollama serve
```

**启动后端服务：**
```powershell
cd agent-hub-backend
.\.venv\Scripts\Activate.ps1
python main.py
```

**启动前端服务：**
```powershell
cd agent-hub-frontend
npm run dev
```

## 八、部署决策总结

### 8.1 决策矩阵

| 评估维度 | 服务器端部署 | 本地部署 | 最终选择 |
|----------|-------------|----------|----------|
| 功能完整性 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 本地部署 |
| 响应速度 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 本地部署 |
| 资源需求 | ⭐⭐ | ⭐⭐⭐⭐ | 本地部署 |
| 部署难度 | ⭐⭐ | ⭐⭐⭐⭐⭐ | 本地部署 |
| 数据安全 | ⭐⭐⭐ | ⭐⭐⭐⭐⭐ | 本地部署 |
| 离线可用 | ⭐ | ⭐⭐⭐⭐⭐ | 本地部署 |

### 8.2 决策理由

1. **功能需求驱动**：项目需要 OCR 和语音识别功能，服务器端部署无法满足
2. **用户体验优先**：本地部署响应更快，无需等待网络传输
3. **资源限制考虑**：本地部署对硬件要求较低，更容易推广使用
4. **数据隐私保护**：本地部署避免数据上传，更符合隐私保护要求
5. **离线使用场景**：支持断网环境下使用，提升系统可靠性

### 8.3 未来优化方向

1. **混合部署优化**：进一步完善服务器端和本地部署的智能切换逻辑
2. **模型升级**：在硬件允许的情况下，升级到更大的模型以提升效果
3. **分布式部署**：探索多节点协同推理方案
4. **性能监控**：添加模型推理性能监控和日志记录
